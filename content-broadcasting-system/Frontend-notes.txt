Content Broadcasting System - Frontend Notes
============================================

This is a React + Vite app. Below are notes I kept while building it so
whoever picks this up (or future me) doesn't have to reverse-engineer everything.


FOLDER LAYOUT
-------------
Pretty standard feature-based structure. Components are split into:
  - ui/         generic stuff like Button, Modal, Badge — no business logic
  - content/    things that know about the content domain (previews, badges, modals)
  - layout/     the sidebar and the dashboard wrapper
  - routing/    just ProtectedRoute

Pages are split by role under pages/teacher/ and pages/principal/.
The live broadcast page lives at pages/LivePage.jsx and is public (no auth).

Services live in src/services/ and that's the only place API calls happen.
Components never call fetch or axios directly — they go through a service.


AUTH
----
Login hits authService.login() which validates against mock data right now.
On success it stores the token in localStorage under "cbs_token" and the user
object under "cbs_user". AuthContext reads that on load so sessions survive a refresh.

The axios instance in api.js has a request interceptor that attaches the Bearer
token automatically, so service functions don't have to think about headers.

ProtectedRoute checks two things: is the user logged in, and do they have the
right role for this route. If the role is wrong it redirects to their own dashboard
instead of showing a 403 or something confusing.


ROUTING
-------
Teacher routes:    /teacher/dashboard, /teacher/upload, /teacher/my-content
Principal routes:  /principal/dashboard, /principal/pending, /principal/all-content
Public:            /login, /live/:teacherId

The live page takes a teacherId param and shows only that teacher's approved
content that's currently within its scheduled time window.


HOOKS
-----
I made two fetch hooks because the use cases are different:

  useFetch   - fires automatically on mount, good for loading page data
  useAsync   - you call execute() manually, good for form submissions and actions

usePagination handles slicing an array client-side. It takes the full list and
a page size, returns the current slice plus navigation helpers. Pagination resets
to page 1 when you call reset() — I wire that up to filter/search changes.


STATE
-----
No Redux, no Zustand. AuthContext for the global user state, everything else is
local useState per page. It's simple enough that global state management would
just add noise.

React.memo on ContentPreview, StatusBadge, ScheduleBadge because those render
inside big lists and the props rarely change.

useMemo in AllContent for the filtered list — 500+ items, don't want to recompute
on every keystroke if the deps haven't changed.


MOCK DATA
---------
mockData.js has 6 hand-written items covering all the interesting cases (active,
scheduled, expired, pending, approved, rejected) plus a loop that generates ~514
more for pagination/performance testing.

The mock data is a plain mutable array at module scope. Approve/reject/upload
mutations happen directly on it, so changes persist for the browser session but
reset on page reload. That's fine for a demo.

To swap in a real backend: set VITE_API_URL in .env and replace the mock logic
in each service file with real axios calls. The hooks and components don't need
to change at all.


LIVE PAGE
---------
Polls every 30 seconds (POLL_INTERVAL constant at the top of LivePage.jsx).
Content rotates automatically based on each item's rotationDuration field.
The rotation is just a setTimeout that advances the activeIndex — nothing fancy.
Manual refresh button does a silent re-fetch (no full loading spinner).


FILE UPLOADS
------------
Accepts JPG, PNG, GIF up to 10MB. Validated client-side before the service call.
In mock mode, uploaded files get a blob URL for preview. In production you'd
send multipart/form-data to a real endpoint and store the returned URL.

KNOWN SHORTCUTS / TODO
----------------------
- Token is in localStorage. Fine for this scope, httpOnly cookie would be better
  in production.
- No real file storage. Blob URLs work for the demo but don't survive a refresh.
- Schedule status (active/scheduled/expired) is computed client-side from
  startTime/endTime. The server should probably own this in production.
- No tests yet. The hooks and service functions are pure enough that unit tests
  would be straightforward to add.
